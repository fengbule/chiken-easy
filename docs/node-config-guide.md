# Node Config Guide

## Supported Protocol Wizards

The current node wizard supports:

- `VMess + WebSocket`
- `VLESS + Reality`
- `Trojan + TLS`
- `Hysteria2`
- `Shadowsocks`
- `Mixed HTTP/SOCKS`

## Wizard Flow

1. Select agent
2. Select protocol
3. Fill protocol-specific fields
4. Optionally set export name and export host
5. Preview config JSON
6. Apply config to the agent

## Protocol Notes

### VMess + WebSocket

- generates UUID and WS path defaults
- suitable for standard WS inbound scenarios

### VLESS + Reality

- requires private key on the server side
- `publicKey` should be filled if the profile will later be exported into subscriptions
- `shortId`, `serverName`, and handshake port are all part of the exported client shape

### Trojan + TLS

- self-signed certs are auto-generated by the agent when missing
- real acceptance traffic was verified through a deployed trojan node

### Hysteria2

- agent can auto-generate self-signed certs when needed
- bandwidth fields are exposed in the wizard

### Shadowsocks

- default method is `aes-256-gcm`
- real acceptance flow was validated previously and remains supported in this branch

### Mixed

- useful as a fast local HTTP/SOCKS mixed proxy for quick tests

## Config Versions and Rollback

Config APIs:

- `GET /api/agents/:id/config`
- `POST /api/agents/:id/config`
- `GET /api/agents/:id/config/versions`
- `POST /api/agents/:id/config/rollback/:versionId`
- `POST /api/agents/:id/config/wizard`

Behavior:

- each apply stores a config version
- apply failures do not erase previous stored versions
- rollback requests are audited

## Forwarding

Forward APIs:

- `POST /api/forward/render`
- `GET /api/agents/:id/forwards`
- `POST /api/agents/:id/forward/wizard`
- `DELETE /api/agents/:id/forwards/:ruleId`

Current forwarding engines:

- `sing-box`
- `realm`
- `gost`

## Real Acceptance Summary

Completed with real servers:

- trojan node applied and produced successful real egress earlier in the acceptance run
- local node profile was imported back into the node pool
- sing-box TCP forwarding created active containers and was reachable
- sing-box UDP forwarding answered real DNS queries
- forwarding cleanup removed containers successfully

Observed limitations during real acceptance:

- `realm` forwarding failed on the remote agent host because the registry pull timed out
- `gost` forwarding failed for the same reason
- these are deployment-environment image-availability failures, not wizard syntax failures
